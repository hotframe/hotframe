# 6. Database Protocols (`db/`)

> The `db/` folder defines the **minimal persistence abstractions** on which the rest of the framework and application modules build their code: formal protocols that decouple the domain from SQLAlchemy, a mixin for per-tenant singleton rows, and column types with transparent encryption.

---

## What this folder is for

A classic mistake in ORM-backed applications: business logic imports `AsyncSession` directly, making it impossible to test without a real database.

hotframe cuts through this problem with three instruments:

1. **Protocols** (`protocols.py`): formal Python `typing.Protocol` interfaces that describe what a session, a query builder, and a repository do — without mentioning SQLAlchemy at any point.
2. **`SingletonMixin`** (`singletons.py`): a get-or-create pattern for configuration models of which exactly one record must exist per tenant (`hub_id`).
3. **Encrypted types** (`types.py`): SQLAlchemy `TypeDecorator` subclasses that encrypt and decrypt using Fernet transparently, without application code being aware of the encryption.

This folder is intentionally small. Its responsibility is the **persistence contract**, not the implementation. The concrete implementation (SQLAlchemy `AsyncSession`, `BaseRepository`, `HubQuery`) lives elsewhere in the framework.

## `protocols.py` — The five persistence protocols

The protocols are written in Python 3.12 with modern generic syntax (`class IQueryBuilder[T](Protocol)`). They are pure `Protocol` types — **not** abstract classes; you do not need to inherit from them; it is enough to satisfy the interface structurally (typed duck typing).

### `IScalarResult[T_co]`

```python
class IScalarResult[T_co](Protocol):
    def all(self) -> list[T_co]: ...
    def first(self) -> T_co | None: ...
```

Represents the result of calling `.scalars()` on an execute result. The `T_co` parameter is covariant in semantics — in practice it is the type of the model being queried.

Methods:
- `all()` → full list of instances.
- `first()` → first instance or `None`.

This is the minimal interface required by `IExecuteResult.scalars()`.

### `IExecuteResult`

```python
class IExecuteResult(Protocol):
    def scalars(self) -> IScalarResult[Any]: ...
    def scalar_one(self) -> Any: ...
    def scalar_one_or_none(self) -> Any | None: ...
    def first(self) -> Any | None: ...
    def all(self) -> list[Any]: ...
```

Wraps the result of `session.execute()`. Maps directly onto the `CursorResult`/`ChunkedIteratorResult` API of SQLAlchemy 2.0, with the difference that being a `Protocol` it can be implemented with any data source.

Key methods:
- `scalars()` → `IScalarResult` for iterating over the first column.
- `scalar_one()` → a single value; raises if there are zero or more than one.
- `scalar_one_or_none()` → a value or `None`; raises if there is more than one.
- `first()` → first row as a tuple or `None`.
- `all()` → all rows.

### `ISession`

```python
class ISession(Protocol):
    async def execute(self, statement: Any, parameters: Any = None) -> IExecuteResult: ...
    def add(self, instance: Any) -> None: ...
    async def flush(self, objects: Any = None) -> None: ...
    async def rollback(self) -> None: ...
    async def commit(self) -> None: ...
    async def delete(self, instance: Any) -> None: ...
    def in_transaction(self) -> bool: ...
    def begin(self) -> Any: ...
    def begin_nested(self) -> Any: ...
```

The central abstraction. Represents an async database session with the minimal set of operations any application code needs:

- `execute` / `add` / `flush` / `commit` / `rollback` / `delete` — the classic transaction lifecycle.
- `in_transaction()` — session state.
- `begin()` / `begin_nested()` — explicit transaction and savepoint management.

The real implementation is `sqlalchemy.ext.asyncio.AsyncSession`. In tests you can pass any object that satisfies this protocol.

In FastAPI routes the recommended injection type is the `DbSession` alias:

```python
from hotframe import DbSession  # = Annotated[ISession, Depends(get_db)]

@router.get("/items")
async def items(db: DbSession):
    result = await db.execute(select(Item))
    return result.scalars().all()
```

### `IQueryBuilder[T]`

```python
class IQueryBuilder[T](Protocol):
    def filter(self, *conditions: Any) -> Self: ...
    def order_by(self, *columns: Any) -> Self: ...
    def options(self, *opts: Any) -> Self: ...
    def limit(self, n: int) -> Self: ...
    def offset(self, n: int) -> Self: ...
    def with_deleted(self) -> Self: ...
    async def all(self) -> list[T]: ...
    async def first(self) -> T | None: ...
    async def get(self, id: UUID) -> T | None: ...
    async def count(self) -> int: ...
    async def sum(self, column: Any) -> Decimal: ...
    async def exists(self) -> bool: ...
    async def get_or_create(self, **defaults: Any) -> tuple[T, bool]: ...
    async def delete(self, id: UUID) -> bool: ...
    async def hard_delete(self, id: UUID) -> bool: ...
```

Interface for a chainable query builder over model `T`. Filtering/pagination methods return `Self` (the same builder, modified); terminal methods are `async`.

Notable methods:
- `with_deleted()` → include soft-deleted records in results (for models using `SoftDeleteMixin`).
- `get_or_create(**defaults)` → returns `(instance, was_created)`.
- `delete(id)` → soft delete (sets `deleted_at`).
- `hard_delete(id)` → permanent physical deletion.
- `sum(column)` → typed aggregate result as `Decimal`.

The real implementation is `HubQuery`, which adds automatic `hub_id` filtering.

### `IRepository[T]`

```python
class IRepository[T](Protocol):
    async def list(
        self,
        *,
        search: str | None = None,
        order_by: Any | None = None,
        limit: int = 50,
        offset: int = 0,
        options: Sequence[Any] | None = None,
        **filters: Any,
    ) -> dict[str, Any]: ...
    async def get(self, id: UUID, *, options: Sequence[Any] | None = None) -> T | None: ...
    async def create(self, **kwargs: Any) -> T: ...
    async def update(self, id: UUID, **kwargs: Any) -> T | None: ...
    async def delete(self, id: UUID) -> bool: ...
    async def hard_delete(self, id: UUID) -> bool: ...
    async def count(self, **filters: Any) -> int: ...
    async def exists(self, **filters: Any) -> bool: ...
```

Full typed CRUD over model `T`. Key differences from `IQueryBuilder`:

- `list()` returns `dict[str, Any]` — a paginated envelope with `items`, `total`, `limit`, `offset`. Not a raw list, but a structure ready to serialize to JSON in an API response.
- `update(id, **kwargs)` by id, not by instance — designed for REST APIs.
- `options` in `list` and `get` allows passing SQLAlchemy relationship loaders (`selectinload`, `joinedload`) without breaking the interface.
- `delete` is soft; `hard_delete` is physical.

The real implementation is `BaseRepository[T]`, documented in section 12 of the main guide.

## `singletons.py` — `SingletonMixin`

```python
class SingletonMixin:
    @classmethod
    async def get_config(cls, session: ISession, hub_id: UUID) -> Self: ...
```

A mixin designed for **per-tenant configuration models**: records of which exactly one must exist per `hub_id`. The pattern is get-or-create:

```python
# Inside singletons.py:
stmt = select(cls).where(cls.hub_id == hub_id).limit(1)
result = await session.execute(stmt)
instance = result.scalars().first()

if instance is not None:
    return instance

instance = cls(hub_id=hub_id)
session.add(instance)
await session.flush()
return instance
```

The `flush()` persists the object within the current transaction without committing — the caller decides whether to confirm. The `limit(1)` is a defensive optimization, although the unique constraint on `hub_id` already guarantees uniqueness.

For concurrency safety, any model using this mixin must have a `UNIQUE` constraint on `hub_id` at the database level. Without it, two concurrent coroutines could create two records before either flushes.

### Canonical usage

```python
from hotframe import Base
from hotframe.db.singletons import SingletonMixin
from hotframe.models.mixins import HubMixin, TimestampMixin
from sqlalchemy.orm import Mapped, mapped_column
from sqlalchemy import String

class ShopConfig(Base, SingletonMixin, HubMixin, TimestampMixin):
    __tablename__ = "shop_config"
    shop_name: Mapped[str] = mapped_column(String(255), default="My Shop")
    currency: Mapped[str] = mapped_column(String(3), default="EUR")

# In a view or service:
config = await ShopConfig.get_config(session, hub_id=request.state.hub_id)
config.shop_name = "New store name"
await session.commit()
```

### Why it accepts `ISession` instead of `AsyncSession`

The `session` parameter is typed as `ISession` (imported under `TYPE_CHECKING`). This keeps the subsystem consistent: the mixin is testable with a fake session without touching the database.

## `types.py` — Columns with transparent encryption

Two SQLAlchemy `TypeDecorator` subclasses that implement Fernet encryption on the way to and from the database. "Transparent" means Python code reads and writes plaintext strings; encryption and decryption happen in the ORM hooks.

### `EncryptedString`

```python
class EncryptedString(TypeDecorator):
    impl = String
    cache_ok = True

    def __init__(self, length: int = 512, **kwargs: Any) -> None: ...
    def process_bind_param(self, value: str | None, dialect: Any) -> str | None: ...
    def process_result_value(self, value: str | None, dialect: Any) -> str | None: ...
```

A `VARCHAR` column that stores Fernet ciphertext. Key points:

- `impl = String` — the underlying column is `VARCHAR`.
- `cache_ok = True` — safe for SQLAlchemy's query cache (the type has no mutable state).
- `length` defaults to `512`. Fernet ciphertext is always longer than the plaintext (IV + HMAC + base64 overhead). The rule of thumb from the source code: `length >= 4 * plaintext_max_length`.
- `process_bind_param` — runs on write: encrypts the string with `encrypt_secret`.
- `process_result_value` — runs on read: decrypts with `decrypt_secret`.
- `None` is preserved as `NULL` in the database — it is neither encrypted nor decrypted.

### `EncryptedText`

```python
class EncryptedText(TypeDecorator):
    impl = Text
    cache_ok = True
    # No custom __init__ — inherits TypeDecorator's default
```

Same as `EncryptedString` but with `impl = Text` (`TEXT` column). For large payloads: PEM certificates, base64-encoded `.pfx` files, long tokens, encrypted JSON.

The absence of a custom `__init__` is correct: `Text` does not accept a `length` argument, so there is no parameter to customize.

### Where `encrypt_secret` and `decrypt_secret` come from

Both functions are imported from `hotframe.auth.crypto`. That module implements Fernet encryption using the `SECRET_KEY` from settings. **The type does not store the key** — it reads it at runtime through `crypto`, which means encryption is activated automatically when `SECRET_KEY` is configured.

### Usage in a model

```python
from hotframe.db.types import EncryptedString, EncryptedText
from hotframe import Base
from sqlalchemy.orm import Mapped, mapped_column

class Integration(Base):
    __tablename__ = "integrations"
    id: Mapped[int] = mapped_column(primary_key=True)
    # The API key is stored encrypted; read back as plaintext
    api_key: Mapped[str] = mapped_column(EncryptedString(length=256))
    # The PEM certificate can be large
    certificate_pem: Mapped[str | None] = mapped_column(EncryptedText, nullable=True)
```

```python
from hotframe.db.protocols import (
    IExecuteResult,

__all__ = [
    "IExecuteResult",
    "IQueryBuilder",
    "IRepository",
    "IScalarResult",
    "ISession",
]
```

Note: `SingletonMixin`, `EncryptedString`, and `EncryptedText` are **not in `db/__init__.py`**. They are imported directly from their modules. The `__init__.py` docstring states this explicitly:

```python
from hotframe.db.singletons import SingletonMixin
from hotframe.db.types import EncryptedString, EncryptedText
```

This is a design decision: the protocols are the most frequently imported interface (virtually every service and repository uses them), while `SingletonMixin` and the encrypted types are more specialized tools.

## How this fits into the rest of the framework

- **`BaseRepository[T]`** (in `hotframe/repository/`): implements `IRepository[T]`. Modules inherit from `BaseRepository` and get CRUD for free. Section 12 of the main guide covers its usage.
- **`HubQuery`**: implements `IQueryBuilder[T]` and adds automatic `hub_id` filtering for multi-tenant models with `HubMixin`.
- **SQLAlchemy's `AsyncSession`**: implements `ISession`. FastAPI's `Depends(get_db)` injects a concrete `AsyncSession`, but the code that receives it types it as `ISession`.
- **`DbSession`** (alias in `hotframe/__init__.py`): `Annotated[ISession, Depends(get_db)]` — the canonical type for route injection.
- **`hotframe.auth.crypto`**: `EncryptedString` and `EncryptedText` depend on this module for Fernet encryption and decryption functions.
- **`HubMixin`** and **`TimestampMixin`** (`hotframe/models/mixins.py`): `SingletonMixin` is designed to be combined with `HubMixin` (provides `hub_id`) and `TimestampMixin` (`created_at`, `updated_at`).
- **Dynamic modules**: modules define their own repositories typed against `IRepository` and `ISession`, which allows them to be tested in isolation without the database.

## Gotchas and design decisions

**1. Protocols are not base classes — they are structural interfaces.**
Do not inherit from `ISession` or `IRepository`. They are `Protocol`, not ABCs. Inheriting from them gives you nothing and can cause confusion. Instead, any class that implements the same methods satisfies the protocol automatically.

**2. `IRepository.list()` returns `dict`, not `list`.**
The signature returns `dict[str, Any]` — a paginated envelope, not a raw list. The real implementation returns `{"items": [...], "total": n, "limit": n, "offset": n}`. Designed for REST APIs; do not use it when you just need to iterate locally (prefer `IQueryBuilder.all()`).

**3. `SingletonMixin` does not commit.**
It only flushes. The transaction remains open for the caller to decide whether to commit alongside other operations. Never call `get_config` assuming the object has been persisted — you must commit explicitly.

**4. `EncryptedString.length` is the ciphertext length, not the plaintext length.**
Fernet ciphertext in base64 is significantly longer than the plaintext (16-byte IV + 8-byte timestamp + 32-byte HMAC-SHA256 + padding, all base64-encoded). The rule of thumb from the source code: `length >= 4 * plaintext_max_length`. For a 64-character token, use `EncryptedString(length=256)`.

**5. Encryption requires `SECRET_KEY` to be configured.**
If `SECRET_KEY` is not in settings, `encrypt_secret`/`decrypt_secret` will fail at runtime when attempting to write or read an encrypted column. There is no error at model definition time — the failure occurs when the query is executed.

**6. `IQueryBuilder.with_deleted()` is part of the protocol.**
Even though not all models use soft deletion, the method is part of the protocol. Implementations for models without `SoftDeleteMixin` simply ignore it (return `self` unchanged). This keeps the protocol uniform and avoids branching in client code.

**7. `db/__init__.py` does not export `SingletonMixin` or the types.**
This is a granularity decision: the protocols are the core contract (everyone uses them), while `SingletonMixin` and the encrypted types are specialized tools. Import directly from `hotframe.db.singletons` and `hotframe.db.types` when you need them.
