# 6. Protocolos de base de datos (`db/`)

> La carpeta `db/` define las **abstracciones mínimas de persistencia** sobre las que el resto del framework y los módulos escriben su código: protocolos formales que desacoplan el dominio de SQLAlchemy, un mixin para filas singleton por tenant, y tipos de columna con cifrado transparente.

---

## Para qué sirve esta carpeta

Un error clásico en aplicaciones con ORM: el código de negocio importa `AsyncSession` directamente, lo que hace imposible testear sin levantar una base de datos real.

hotframe corta este problema con tres instrumentos:

1. **Protocolos** (`protocols.py`): interfaces formales de Python (`typing.Protocol`) que describen qué hace una sesión, un query builder y un repositorio — sin mencionar SQLAlchemy en ningún momento.
2. **`SingletonMixin`** (`singletons.py`): patrón get-or-create para modelos de configuración de los que debe existir exactamente un registro por tenant (`hub_id`).
3. **Tipos cifrados** (`types.py`): `TypeDecorator` de SQLAlchemy que cifran y descifran en Fernet de forma transparente, sin que el código de la aplicación sepa que hay cifrado.

Esta carpeta es deliberadamente pequeña. Su responsabilidad es el **contrato de persistencia**, no la implementación. La implementación concreta (SQLAlchemy `AsyncSession`, `BaseRepository`, `HubQuery`) vive en otras partes del framework.

## `protocols.py` — Los cinco protocolos de persistencia

Los protocolos están escritos en Python 3.12 con la sintaxis genérica moderna (`class IQueryBuilder[T](Protocol)`). Son `Protocol` puros — **no** son clases abstractas, no hace falta heredar de ellos; basta con satisfacer la interfaz estructuralmente (duck typing tipado).

### `IScalarResult[T_co]`

```python
class IScalarResult[T_co](Protocol):
    def all(self) -> list[T_co]: ...
    def first(self) -> T_co | None: ...
```

Representa el resultado de llamar `.scalars()` sobre un execute result. El parámetro `T_co` es covariante en la semántica — en la práctica es el tipo del modelo que se está consultando.

Métodos:
- `all()` → lista completa de instancias.
- `first()` → primera instancia o `None`.

Es la interfaz mínima que necesita `IExecuteResult.scalars()`.

### `IExecuteResult`

```python
class IExecuteResult(Protocol):
    def scalars(self) -> IScalarResult[Any]: ...
    def scalar_one(self) -> Any: ...
    def scalar_one_or_none(self) -> Any | None: ...
    def first(self) -> Any | None: ...
    def all(self) -> list[Any]: ...
```

Wrappea el resultado de `session.execute()`. Mapea directamente sobre la API de `CursorResult`/`ChunkedIteratorResult` de SQLAlchemy 2.0, con la diferencia de que al ser un `Protocol` se puede implementar con cualquier origen de datos.

Métodos importantes:
- `scalars()` → `IScalarResult` para iterar por la primera columna.
- `scalar_one()` → un único valor; lanza si hay cero o más de uno.
- `scalar_one_or_none()` → un valor o `None`; lanza si hay más de uno.
- `first()` → primera fila como tupla o `None`.
- `all()` → todas las filas.

### `ISession`

```python
class ISession(Protocol):
    async def execute(self, statement: Any, parameters: Any = None) -> IExecuteResult: ...
    def add(self, instance: Any) -> None: ...
    async def flush(self, objects: Any = None) -> None: ...
    async def rollback(self) -> None: ...
    async def commit(self) -> None: ...
    async def delete(self, instance: Any) -> None: ...
    def in_transaction(self) -> bool: ...
    def begin(self) -> Any: ...
    def begin_nested(self) -> Any: ...
```

La abstracción central. Representa una sesión de base de datos asíncrona con el conjunto mínimo de operaciones que cualquier código de aplicación necesita:

- `execute` / `add` / `flush` / `commit` / `rollback` / `delete` — el ciclo de vida clásico de una transacción.
- `in_transaction()` — estado de la sesión.
- `begin()` / `begin_nested()` — gestión explícita de transacciones y savepoints.

La implementación real es `sqlalchemy.ext.asyncio.AsyncSession`. En tests se puede pasar cualquier objeto que satisfaga este protocolo.

En las rutas FastAPI el tipo de inyección recomendado es el alias `DbSession`:

```python
from hotframe import DbSession  # = Annotated[ISession, Depends(get_db)]

@router.get("/items")
async def items(db: DbSession):
    result = await db.execute(select(Item))
    return result.scalars().all()
```

### `IQueryBuilder[T]`

```python
class IQueryBuilder[T](Protocol):
    def filter(self, *conditions: Any) -> Self: ...
    def order_by(self, *columns: Any) -> Self: ...
    def options(self, *opts: Any) -> Self: ...
    def limit(self, n: int) -> Self: ...
    def offset(self, n: int) -> Self: ...
    def with_deleted(self) -> Self: ...
    async def all(self) -> list[T]: ...
    async def first(self) -> T | None: ...
    async def get(self, id: UUID) -> T | None: ...
    async def count(self) -> int: ...
    async def sum(self, column: Any) -> Decimal: ...
    async def exists(self) -> bool: ...
    async def get_or_create(self, **defaults: Any) -> tuple[T, bool]: ...
    async def delete(self, id: UUID) -> bool: ...
    async def hard_delete(self, id: UUID) -> bool: ...
```

Interfaz de un query builder encadenable para el modelo `T`. Los métodos de filtrado/paginación devuelven `Self` (el mismo builder modificado), los terminales son `async`.

Métodos destacados:
- `with_deleted()` → incluye registros con soft-delete en los resultados (para modelos que usan `SoftDeleteMixin`).
- `get_or_create(**defaults)` → devuelve `(instancia, fue_creada)`.
- `delete(id)` → soft-delete (pone `deleted_at`).
- `hard_delete(id)` → eliminación física permanente.
- `sum(column)` → suma tipada como `Decimal`.

La implementación real es `HubQuery` que incorpora filtrado automático por `hub_id`.

### `IRepository[T]`

```python
class IRepository[T](Protocol):
    async def list(
        self,
        *,
        search: str | None = None,
        order_by: Any | None = None,
        limit: int = 50,
        offset: int = 0,
        options: Sequence[Any] | None = None,
        **filters: Any,
    ) -> dict[str, Any]: ...
    async def get(self, id: UUID, *, options: Sequence[Any] | None = None) -> T | None: ...
    async def create(self, **kwargs: Any) -> T: ...
    async def update(self, id: UUID, **kwargs: Any) -> T | None: ...
    async def delete(self, id: UUID) -> bool: ...
    async def hard_delete(self, id: UUID) -> bool: ...
    async def count(self, **filters: Any) -> int: ...
    async def exists(self, **filters: Any) -> bool: ...
```

CRUD completo tipado sobre el modelo `T`. Diferencias clave con `IQueryBuilder`:

- `list()` devuelve `dict[str, Any]` — un envelope paginado con `items`, `total`, `limit`, `offset`. No una lista raw, sino la estructura lista para serializar a JSON en una API.
- `update(id, **kwargs)` por id, no por instancia — pensado para APIs REST.
- `options` en `list` y `get` permite pasar cargadores relacionales de SQLAlchemy (`selectinload`, `joinedload`) sin romper la interfaz.
- `delete` es soft; `hard_delete` es físico.

La implementación real es `BaseRepository[T]` documentada en la sección 12 de la guía principal.

## `singletons.py` — `SingletonMixin`

```python
class SingletonMixin:
    @classmethod
    async def get_config(cls, session: ISession, hub_id: UUID) -> Self: ...
```

Mixin pensado para modelos de **configuración por tenant**: registros de los que debe existir exactamente uno por `hub_id`. El patrón es get-or-create:

```python
# En singletons.py:
stmt = select(cls).where(cls.hub_id == hub_id).limit(1)
result = await session.execute(stmt)
instance = result.scalars().first()

if instance is not None:
    return instance

instance = cls(hub_id=hub_id)
session.add(instance)
await session.flush()
return instance
```

El `flush()` persiste el objeto en la misma transacción sin hacer `commit` — el llamador decide si confirma. El `limit(1)` es una optimización defensiva aunque el unique constraint en `hub_id` garantiza unicidad.

Para que sea concurrencia-safe, el modelo que use este mixin debe tener una restricción `UNIQUE` en `hub_id` a nivel de base de datos. Sin esa restricción, dos corrutinas concurrentes podrían crear dos registros antes de que alguna haga flush.

### Uso canónico

```python
from hotframe import Base
from hotframe.db.singletons import SingletonMixin
from hotframe.models.mixins import HubMixin, TimestampMixin
from sqlalchemy.orm import Mapped, mapped_column
from sqlalchemy import String

class ShopConfig(Base, SingletonMixin, HubMixin, TimestampMixin):
    __tablename__ = "shop_config"
    shop_name: Mapped[str] = mapped_column(String(255), default="My Shop")
    currency: Mapped[str] = mapped_column(String(3), default="EUR")

# En una vista o servicio:
config = await ShopConfig.get_config(session, hub_id=request.state.hub_id)
config.shop_name = "Nueva tienda"
await session.commit()
```

### Por qué acepta `ISession` y no `AsyncSession`

El parámetro `session` está tipado como `ISession` (importado en `TYPE_CHECKING`). Esto mantiene la coherencia del subsistema: el mixin es testeable con un fake de sesión sin tocar la BD.

## `types.py` — Columnas con cifrado transparente

Dos `TypeDecorator` de SQLAlchemy que implementan cifrado Fernet en tránsito hacia y desde la base de datos. "Transparente" significa que el código Python lee y escribe strings en claro; el cifrado/descifrado ocurre en los hooks del ORM.

### `EncryptedString`

```python
class EncryptedString(TypeDecorator):
    impl = String
    cache_ok = True

    def __init__(self, length: int = 512, **kwargs: Any) -> None: ...
    def process_bind_param(self, value: str | None, dialect: Any) -> str | None: ...
    def process_result_value(self, value: str | None, dialect: Any) -> str | None: ...
```

Columna `VARCHAR` que almacena ciphertext Fernet. Puntos importantes:

- `impl = String` — la columna subyacente es `VARCHAR`.
- `cache_ok = True` — safe para query cache de SQLAlchemy (el tipo no tiene estado mutable).
- `length` por defecto es `512`. El ciphertext Fernet es siempre mayor que el plaintext (overhead de IV + HMAC + base64). La regla del comentario del código: `length >= 4 * plaintext_max_length`.
- `process_bind_param` — se ejecuta al escribir: cifra el string con `encrypt_secret`.
- `process_result_value` — se ejecuta al leer: descifra con `decrypt_secret`.
- `None` se preserva como `NULL` en la BD — no se cifra ni descifra.

### `EncryptedText`

```python
class EncryptedText(TypeDecorator):
    impl = Text
    cache_ok = True
    # Sin __init__ propio — hereda el default de TypeDecorator
```

Igual que `EncryptedString` pero con `impl = Text` (columna `TEXT`). Para payloads grandes: certificados PEM, ficheros `.pfx` en base64, tokens largos, JSON cifrado.

La ausencia de `__init__` propio es correcta: `Text` no acepta `length`, así que no hay parámetro que personalizar.

### De dónde vienen `encrypt_secret` y `decrypt_secret`

Ambas funciones se importan de `hotframe.auth.crypto`. Ese módulo implementa el cifrado Fernet usando la `SECRET_KEY` del settings. **El tipo no guarda la clave** — la lee en tiempo de ejecución a través de `crypto`, lo que significa que el cifrado se activa automáticamente cuando `SECRET_KEY` está configurada.

### Uso en un modelo

```python
from hotframe.db.types import EncryptedString, EncryptedText
from hotframe import Base
from sqlalchemy.orm import Mapped, mapped_column

class Integration(Base):
    __tablename__ = "integrations"
    id: Mapped[int] = mapped_column(primary_key=True)
    # La clave de API se guarda cifrada; se lee en claro
    api_key: Mapped[str] = mapped_column(EncryptedString(length=256))
    # El certificado PEM puede ser grande
    certificate_pem: Mapped[str | None] = mapped_column(EncryptedText, nullable=True)
```

```python
from hotframe.db.protocols import (

__all__ = [
    "IExecuteResult",
    "IQueryBuilder",
    "IRepository",

```

Nota: `SingletonMixin`, `EncryptedString` y `EncryptedText` **no están en el `__init__.py` de `db/`**. Se importan directamente desde sus módulos. La guía del `__init__.py` lo indica explícitamente:

```python
from hotframe.db.singletons import SingletonMixin
from hotframe.db.types import EncryptedString, EncryptedText
```

Esto es una decisión de diseño: los protocolos son la interfaz más frecuentemente importada (prácticamente en todos los servicios y repos), mientras que `SingletonMixin` y los tipos cifrados son de uso más específico.

## Cómo encaja con el resto del framework

- **`BaseRepository[T]`** (en `hotframe/repository/`): implementa `IRepository[T]`. Los módulos heredan de `BaseRepository` y obtienen CRUD gratis. La sección 12 de la guía maestra cubre su uso.
- **`HubQuery`**: implementa `IQueryBuilder[T]` y añade filtrado automático por `hub_id` para modelos multi-tenant con `HubMixin`.
- **`AsyncSession` de SQLAlchemy**: implementa `ISession`. El `Depends(get_db)` de FastAPI inyecta una `AsyncSession` concreta, pero el código que la recibe la tipa como `ISession`.
- **`DbSession`** (alias en `hotframe/__init__.py`): `Annotated[ISession, Depends(get_db)]` — el tipo canónico para inyección en rutas.
- **`hotframe.auth.crypto`**: `EncryptedString` y `EncryptedText` dependen de este módulo para las funciones de cifrado/descifrado Fernet.
- **`HubMixin`** y **`TimestampMixin`** (`hotframe/models/mixins.py`): `SingletonMixin` está diseñado para combinarse con `HubMixin` (aporta `hub_id`) y `TimestampMixin` (`created_at`, `updated_at`).
- **Módulos dinámicos**: los módulos definen sus propios repositorios tipando contra `IRepository` e `ISession`, lo que permite testarlos de forma aislada sin la BD.

---

## Gotchas y decisiones de diseño

**1. Los protocolos no son clases base — son interfaces estructurales**
No heredes de `ISession` ni de `IRepository`. Son `Protocol`, no ABCs. Heredar de ellos no te da nada y puede crear confusión. En cambio, cualquier clase que implemente los mismos métodos satisface el protocolo automáticamente.

**2. `IRepository.list()` devuelve `dict`, no `list`**
La firma devuelve `dict[str, Any]` — un envelope paginado, no una lista cruda. La implementación real devuelve `{"items": [...], "total": n, "limit": n, "offset": n}`. Diseñado para APIs REST; no lo uses cuando solo necesitas iterar localmente (prefiere `IQueryBuilder.all()`).

**3. `SingletonMixin` no hace `commit`**
Solo hace `flush`. La transacción queda abierta para que el llamador decida si confirmar junto con otras operaciones. Nunca llames a `get_config` asumiendo que el objeto persistió — debes hacer `commit` explícitamente.

**4. `EncryptedString.length` es del ciphertext, no del plaintext**
El ciphertext Fernet en base64 es significativamente más largo que el plaintext (overhead de IV de 16 bytes + timestamp de 8 bytes + HMAC-SHA256 de 32 bytes + padding, todo en base64). La regla práctica del código fuente: `length >= 4 * plaintext_max_length`. Para un token de 64 chars, usa `EncryptedString(length=256)`.

**5. El cifrado requiere `SECRET_KEY` configurada**
Si `SECRET_KEY` no está en settings, `encrypt_secret`/`decrypt_secret` fallarán en runtime al intentar escribir o leer una columna cifrada. No hay error en la definición del modelo — el fallo ocurre al ejecutar la query.

**6. `IQueryBuilder.with_deleted()` es parte del protocolo**
Aunque no todos los modelos usan soft-delete, el método forma parte del protocolo. Las implementaciones de modelos sin `SoftDeleteMixin` simplemente lo ignoran (devuelven `self` sin cambio). Esto mantiene el protocolo uniforme y evita bifurcaciones en el código cliente.

**7. `db/__init__.py` no exporta `SingletonMixin` ni los tipos**
Es una decisión de granularidad: los protocolos son el contrato core (todo el mundo los usa), mientras que `SingletonMixin` y los tipos cifrados son herramientas específicas. Importa directamente desde `hotframe.db.singletons` y `hotframe.db.types` cuando los necesites.
